<objective>
Teach the swarm to prove its own work by running tests and scanning for security issues, then feeding failures back into the revision loop.

Purpose: a PR that only “looks right” is not good enough; this phase turns the swarm into a validator, not just a generator.
Output: sandboxed test execution, security findings, and a structured failure-to-revision path.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| generated diff → test runner | Untrusted code can try to escape the sandbox or manipulate outputs. |
| generated diff → security scan | Malicious patterns may be intentionally hidden in code or strings. |
| validator output → orchestrator | False success or noisy failures can mislead the rest of the run. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-02-01 | Elevation of Privilege | test runner | mitigate | Restrict execution to the approved sandbox and explicit test commands only. |
| T-02-02 | Tampering / Spoofing | security auditor | mitigate | Use deterministic rules plus explicit heuristics, and require findings to reference concrete code patterns. |
| T-02-03 | Information Disclosure | validation results | mitigate | Keep stdout/stderr structured and avoid leaking secrets into the dashboard or logs. |
</threat_model>

<verification>
